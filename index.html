architectureThree-layer observability

KleinProbe provides a missing intermediate abstraction between global device calibration and your experiment results.

LAYER 1
backend.properties()
Global device state. All qubits. Updated ~daily. Does not condition on which qubits your circuit uses.
IBM provides
LAYER 2
KleinProbe
Circuit-conditioned snapshot. Executes a structured probe on the same physical qubits selected by the transpiler.
this module
LAYER 3
Your experiment
Actual results. You can now identify which jobs ran under degraded hardware conditions.
you run

The probe uses a fixed structured stabilizer circuit derived from a non-orientable topological construction. Its syndrome output is analytically predictable — the topology defines what should happen; the hardware produces what does happen; the deviation characterises the effective noise on those specific qubits at that moment.

KleinProbe does not modify hardware, error correction schemes, or execution paths. It is a diagnostic layer that runs alongside quantum experiments.

Non-goals: KleinProbe does not perform transpiler optimization, error correction, or autonomous execution decisions. It does not predict circuit fidelity. It does not replace backend.properties() — it complements it. The policy layer (opt-in) exposes interfaces for future decision logic; no decisions are made automatically.

metricsWhat KleinProbe measures

Five circuit-conditioned metrics. Z_raw and S serve distinct roles and are not interchangeable.

metricmeaningsensitive to
H Shannon entropy of syndrome distribution (bits) Total noise — more noise = higher H. Increases with T2 decay and gate errors.
inv Klein invariant fraction — P(syndrome bit 0 = 1) Quality of the qubit hosting the antipodal edge. Sensitive to RO and T2 of that specific qubit.
f Dominant pattern frequency How strongly the hardware reproduces the predicted syndrome pattern.
Z_raw Statistical significance vs uniform baseline — (f − p₀) / σ Statistical test statistic. Suitable for significance testing and hypothesis testing.
S Probe signal score — clip(Z_raw / 50, 0, 1) Engineering indicator. S = 1.0 strong signal. S < 0.5 degraded. Not interchangeable with Z_raw.

All metrics reflect the physical qubits selected by the transpiler for your circuit — not the device average.

v0.4.0KleinAtlas — automatic tile discovery

KleinAtlas reads backend.properties() and automatically discovers calibration-aware spatial tiles. No hardcoded patches. Rebuilds when the chip recalibrates.

# Build atlas from current calibration — automatic, no hardcoded patches
from kleinprobe import KleinAtlas

atlas = KleinAtlas(backend)
atlas.build()
print(atlas.report())
# KleinAtlas — ibm_fez
# Calibration: 2026-07-08T16:03:08Z
# Stale: no
# Tile 0 central T2=87.0µs RO=0.0100
# Tile 1 upper-left T2=140.0µs RO=0.0082 ← best T2, worst execution
# Tile 2 middle T2=85.0µs RO=0.0120

# Check if calibration changed — rebuild if stale
if atlas.is_stale():
    atlas.build()

# Transpile probe for each tile — initial_layout ready to use
for tile in atlas.tiles:
    layout = tile.initial_layout(probe_qc)
    pm = generate_preset_pass_manager(..., initial_layout=layout)

Runtime: ~4s for 3-tile probe · zero additional queue time · one job. Atlas rebuilds automatically when is_stale() returns True.

v0.4.0Three-state validity model

KleinProbe measurements have three states. Only INVALID warrants discarding — VALID_ANOMALOUS is a successful detection, not a failure.

from kleinprobe import classify_tiled, VALID_SPATIAL, VALID_ANOMALOUS, INVALID

validity = classify_tiled(tile_data, backend='ibm_marrakesh')
print(validity.execution_map('ibm_marrakesh'))

v0.3.0Tiled spatial probe

Run multiple probe circuits simultaneously across non-overlapping chip regions. Returns per-region H and inv — a sampled spatial execution-state field. Hardware validated on ibm_fez: H spread = 0.93 bits across 3 regions.

# 3-tile spatial probe — ibm_fez hardware validated 2026-07-07
from kleinprobe.tiling import TiledSnapshot, SpatialHardwareState

tsnap = probe.run_tiled(n_tiles=3, seeds=[39, 2, 175])

if tsnap.is_valid:
    print(tsnap.report())
    # Tile 0: H=3.78 inv=0.820 ✓ (central region)
    # Tile 1: H=4.70 inv=0.729 ✓ (upper-left)
    # Tile 2: H=4.71 inv=0.728 ✓ (middle)
    # Spatial variance: 0.532

spatial = SpatialHardwareState.from_tiled(tsnap)
spatial.H_spread # 0.9335 bits — significant spatial variation
spatial.inv_spread # 0.0918 — qubit quality varies by region
spatial.spatial_variance # scalar uniformity measure

Runtime: ~3s single probe (δ=0) · ~4s 3-tile probe · zero additional queue time (PUB mode). The previous ~7s figure included the full δ-family (4 PUBs).

v0.3.0 · lmsLayout Match Score

Before running KleinProbe, check how well the probe layout matches your target circuit's physical qubit placement. A high score means the probe is measuring the same chip region your experiment uses.

from kleinprobe import layout_match_score

match = layout_match_score(probe_isa, target_isa)
print(match.report())
# Layout Match Score
# Probe relevance: 0.833 (15/18 qubits shared) [HIGH]
# Target coverage: 0.300 (15/50 target qubits)
# Shared qubits: [87, 88, 89, 90, 91, ...]

match.lms_probe # 0-1: probe relevance to your circuit
match.lms_target # 0-1: fraction of your circuit characterised
match.relevance # 'HIGH' | 'MODERATE' | 'LOW' | 'NEGLIGIBLE'

# With tiling — find which tile best matches your circuit
best = tsnap.best_match(target_isa)
# {'tile': 0, 'lms_probe': 0.944, 'H': 3.78, 'relevance': 'HIGH'}

Called Layout Match Score (not "overlap") to avoid confusion with tile-to-tile disjointness, which is the opposite concept.

formalismStatistical definition

KleinProbe has a formal statistical identity published as a citable paper. The paper defines the latent hardware state Θ(L,t), proves consistency of the estimator under stationarity, and positions the method relative to randomised benchmarking, noise spectroscopy, and just-in-time transpilation.

Zenodo · 2026
KleinProbe: A Circuit-Conditioned Statistical Estimator for Effective Quantum Hardware Noise
doi:10.5281/zenodo.21186259  ·  Leonardo Roma  ·  MIT

The estimator is defined as θ̂KP(L,t) = (H, I, f, Zraw, S) ∈ ℝ⁵, mapping a transpiled layout and timestamp to a five-dimensional summary of the latent hardware state Θ(L,t). Future work can cite this quantity directly when reporting circuit-conditioned hardware conditions.
